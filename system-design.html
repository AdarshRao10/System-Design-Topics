Why bother understanding this at all?

Most engineers meet "system design" as an interview format a whiteboard, a 45-minute clock, a request to design something Twitter-shaped. That framing makes it feel like trivia: memorize the load balancer, memorize the cache, recite it back.

It isn't trivia. It's the difference between a product that degrades gracefully and one that falls over completely the first time it gets popular. The good news is that almost everything below is built from the same handful of ideas, reapplied at different scales.

This first part stays close to the ground: what a request actually is, where it comes from, and what happens to it the moment more than one person sends one at the same time. Parts two through four build caching, asynchronous messaging, and global distribution on top of this foundation.

Once resolved, the request lands from one of two doors a browser or a mobile app and the two doors want completely different things.

A browser wants the whole page. Chrome, Safari, Firefox none of them have any idea what your product looks like until your server tells them, in HTML, CSS, and JavaScript.

  • Full layout required the server ships markup and styling, not just facts.
  • Cookies & sessions login state usually rides in a cookie.
  • Stable pipe assumed Wi-Fi or broadband, so bigger payloads are fine.
  • Always current refresh and you get the latest version, no versioning question.

A mobile app already has the layout. It was compiled into the app weeks ago. All it wants from your server is raw facts.

  • Just JSON API calls in, structured data out.
  • Token-based auth JWTs, not cookies.
  • Version support someone's always on last year's build.
  • Flaky pipe assumed 5G one minute, a basement the next.

02

Splitting the load: web tier vs. data tier

This setup works fine for a modest user base but it doesn't survive real traffic, because not everything strains at the same rate.

As users grow, the part that answers requests and the part that stores data start to feel pressure differently so it pays to separate them on purpose.

  • Web tier handles incoming requests, returns HTML or JSON. Add two or three more servers behind a load balancer. Cheap, fast, repeatable.
  • Data tier the database. Scaling it means read replicas or sharding. Keeping the two tiers separate means you grow the web layer without ever touching the database.

03

Choosing a database: relational vs. non-relational

"But which database?" The honest answer is: it depends on whether your data wants to be a spreadsheet or a folder.

Relational

PostgreSQL · MySQL · SQLite · Oracle

Non-Relational

Cassandra · Redis · Neo4j · MongoDB

Relational databases are a perfectly organized spreadsheet: fixed tables, fixed columns, clear relationships between them. That rigidity buys you strong consistency exactly what you want for a bank balance or a checkout flow.

Non-relational databases are an open folder: documents, key-value pairs, graphs whatever shape the data actually has. Most are built to scale horizontally with very low read/write latency, which is why they show up under social feeds and caching layers.

RULE OF THUMB → Fixed shape + 100% accuracy required → relational. Flexible shape + huge volume + raw speed → non-relational.

Scaling up vs. scaling out

"Code is easy. Keeping it alive when a million people show up at once that's the actual job." There are exactly two ways to respond to that pressure.

Vertical make it bigger
8GB RAM → 64GB RAM, same single box
Horizontal make more of it
same-size boxes, cloned on demand

Vertical scaling is turning your old computer into a superhero: same machine, bigger engine. The catch is a hard ceiling eventually no machine is big enough, and that one machine is still a single point of failure.

05

The load balancer: traffic's police officer

10,000 people hit your site at once, and you're running three servers. Someone has to direct traffic that's the load balancer's entire job.

It sits at the very front, catches everything coming in, and hands it off using one of a few simple algorithms:

  • Round Robin straight down the line: A, then B, then C, then back to A.
  • Least Connections whichever server is least busy right now gets the next request.
  • IP Hash a given user's IP always lands on the same server.

It also runs health checks constantly. Common software options are Nginx and HAProxy; F5 is a well-known hardware option; cloud providers like AWS will run it for you if you'd rather not configure it.

06

The hidden cracks: single points of failure

Look closely at the diagram above and two things stand out: everything depends on that one load balancer, and every server depends on that one database. Kill either, and the whole product goes down.

A single point of failure (SPOF) is any one component that, if it disappears, takes the entire system with it. The fix isn't a clever algorithm it's refusing to let anything important exist in a quantity of one.

07

Redundancy: always have a backup engine

A commercial airplane always flies with two engines. If one fails mid-flight, the other lands the plane safely on its own. Apply the same logic to your load balancer.

ACTIVE
STANDBY

Run two load balancers instead of one. Traffic flows to the active one as normal; the passive one watches and waits. The moment the active balancer crashes, the standby takes over instantly.

08

Health checks: the heartbeat monitor

Having a backup is useless if nothing knows when to use it. The system needs to constantly ask each part: "Are you still alive?"

This is exactly like an ICU heart-rate monitor: a steady beep confirming the patient is stable, and an instant alarm the second something changes. Load balancers ping every server on a tight loop, removing any that stop answering from the rotation until it recovers.

09

Self-healing systems: the lizard's tail

When a lizard loses its tail, it doesn't wait for a vet it just grows another one. Modern cloud infrastructure does the same thing.

The moment a health check flags a dead server, automation AWS Auto Scaling and similar tools terminates the broken instance and spins up a fresh one in its place, usually within seconds, with no engineer touching a keyboard.
