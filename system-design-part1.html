Why bother understanding this at all?

Most engineers meet "system design" as an interview format a whiteboard, a 45-minute clock, a request to design something Twitter-shaped. That framing makes it feel like trivia: memorize the load balancer, memorize the cache, recite it back.

It isn't trivia. It's the difference between a product that degrades gracefully and one that falls over completely the first time it gets popular. The good news is that almost everything below is built from the same handful of ideas, reapplied at different scales. Learn the idea once, and you'll recognize it in every system you read about afterward your bank's app, your group chat, the show you streamed last night.

This first part stays close to the ground: what a request actually is, where it comes from, and what happens to it the moment more than one person sends one at the same time. Parts two through four build caching, asynchronous messaging, and global distribution on top of this foundation in that order, because that's the order real systems hit these walls.

01

Where does traffic even come from?

Every request hits your server from one of two doors a browser or a mobile app and the two doors want completely different things.

A browser wants the whole page. Chrome, Safari, Firefox none of them have any idea what your product looks like until your server tells them, in HTML, CSS, and JavaScript. The server is doing layout work, not just data work.

  • Full layout required the server ships markup and styling, not just facts.
  • Cookies & sessions login state usually rides in a cookie.
  • Assumes a stable pipe most browsers sit on Wi-Fi or broadband, so bigger payloads are fine.
  • Always current refresh the page and you get the latest version, no question of which "build" you're on.

A mobile app already has the layout. It was compiled into the app and installed on the phone weeks ago. All it wants from your server is raw facts.

  • Just JSON API calls in, structured data out. No HTML needed.
  • Token-based auth JWTs and similar, not cookies.
  • Version support someone is always still on last year's build, so your API has to serve multiple versions at once.
  • Assumes a flaky pipe 5G one minute, a basement elevator the next, so payloads stay lightweight.

02

Splitting the load: web tier vs. data tier

This setup works fine for a modest user base but it doesn't survive real traffic, because not everything strains at the same rate.

As users grow, the part that answers requests and the part that stores data start to feel pressure differently so it pays to separate them on purpose.

  • Web tier handles incoming requests, returns HTML or JSON. When traffic spikes, you add two or three more servers behind a load balancer. Cheap, fast, repeatable.
  • Data tier the database. Scaling it is a different exercise entirely: read replicas, sharding, careful migrations. Keeping the two tiers separate means you can grow the web layer freely without ever touching the database.

03

Choosing a database: relational vs. non-relational

"But which database?" The honest answer is: it depends on whether your data wants to be a spreadsheet or a folder.

Relational

PostgreSQL · MySQL · SQLite · Oracle

Non-Relational

Cassandra · Redis · Neo4j · MongoDB

Relational databases are a perfectly organized spreadsheet: fixed tables, fixed columns, clear relationships between them. That rigidity buys you strong consistency exactly what you want for a bank balance or a checkout flow, where "probably correct" isn't good enough.

Non-relational databases are an open folder: documents, key-value pairs, graphs whatever shape the data actually has. No rigid schema means new fields don't require a migration, and most of them are built to scale horizontally with very low read/write latency, which is why they show up under social feeds and caching layers.

RULE OF THUMB → Fixed shape + 100% accuracy required → relational. Flexible shape + huge volume + raw speed → non-relational.
04

Scaling up vs. scaling out

"Code is easy. Keeping it alive when a million people show up at once that's the actual job." There are exactly two ways to respond to that pressure.

Vertical make it bigger
8GB RAM → 64GB RAM, same single box
Horizontal make more of it
same-size boxes, cloned on demand

Vertical scaling is turning your old computer into a superhero: same machine, bigger engine. Almost no code changes you just buy a beefier instance from your cloud provider. The catch is a hard ceiling: eventually no machine on the market is big enough, and that one machine is still a single point of failure.

Horizontal scaling is buying more ordinary cars instead of one giant bus. You don't make any single server bigger; you add more of them. Scalability becomes close to infinite, and losing one server doesn't take the system down but now you need something standing in front of all of them, dividing the traffic. Which brings us to

The load balancer: traffic's police officer

10,000 people hit your site at once, and you're running three servers. Send everyone to server one and it crashes, while the other two sit idle. Someone has to direct traffic that's the load balancer's entire job.

LB
Server A
Server B
Server C

It sits at the very front, catches everything coming in, and hands it off using one of a few simple algorithms:

  • Round Robin straight down the line: A, then B, then C, then back to A.
  • Least Connections whichever server is least busy right now gets the next request.
  • IP Hash a given user's IP always lands on the same server, useful for keeping a session in one place.

It also runs health checks constantly pinging every server to confirm it's actually alive. Common software options are Nginx and HAProxy; F5 is a well-known hardware option; and if you'd rather not configure any of it yourself, cloud providers like AWS will run it for you.

The hidden cracks: single points of failure

Look closely at the diagram above and two things stand out: everything depends on that one load balancer, and every server depends on that one database. Kill either, and the whole product goes down even if every other component is perfectly healthy.

LB

A single point of failure (SPOF) is any one component that, if it disappears, takes the entire system with it. The fix isn't a clever algorithm it's refusing to let anything important exist in a quantity of one.

07

Redundancy: always have a backup engine

A commercial airplane always flies with two engines. If one fails mid-flight, the other lands the plane safely on its own. Apply the same logic to your load balancer.

LB Active
LB Standby

Run two load balancers instead of one. Traffic flows to the active one as normal; the passive one watches and waits. The moment the active balancer crashes, the standby takes over instantly, with no human paged at 3am to fix it.

08

Health checks: the heartbeat monitor

Having a backup is useless if nothing knows when to use it. The system needs to constantly ask each part: "Are you still alive?"

This is exactly like an ICU heart-rate monitor: a steady beep confirming the patient is stable, and an instant alarm the second something changes. Load balancers do the same thing to every server behind them pinging on a tight loop, and quietly removing any server that stops answering from the rotation until it recovers.

09

Self-healing systems: the lizard's tail

When a lizard loses its tail, it doesn't wait for a vet it just grows another one. Modern cloud infrastructure does the same thing.

INSTANCE

The moment a health check flags a dead server, automation AWS Auto Scaling and similar tools terminates the broken instance and spins up a fresh one in its place, usually within seconds, with no engineer touching a keyboard.
